# 運用ルール

## 運用主体

このWikiは、主にChatGPTが会話内容や既存ページをもとに、かなり勝手にページを作成・追記・リンクする。

人間は方針の指定、事実確認、削除や統合などの大きな判断を担当する。内容には雑学的な脱線や未整理のアイデアを含めてよいが、外部世界に関する事実を推測で埋めてはならない。

## 文体

- 本文は原則として「だ・である調」で統一する
- 箇条書きは体言止めまたは常体で統一する
- 引用、製品名、依頼例、コード、外部資料の原文は必要に応じて原文の文体を維持する
- 新規作成・追記時だけでなく、既存ページを編集する際にも周辺の文体を確認する
- 「です・ます調」と「だ・である調」が混在している場合は、意味を変えずに「だ・である調」へ修正する

## 調査と出典

- 商品、人物、企業、歴史、科学、医療、制度、仕様、価格などの外部事実は、書く前に必ず調査する
- 可能な限り、公式サイト、公的機関、一次資料、公式ドキュメントを優先する
- 調査していない内容を、もっともらしい一般論で補完しない
- 確認できなかった内容は断定せず、「未確認」「本人談」「推測」などの区別を明記する
- 医療・健康・法律・金融に関する内容は、商品説明と科学的根拠を混同しない
- 雑学ページにも、原則として「出典」節を設ける
- 情報が後から変わり得る場合は、必要に応じて確認日を記載する

## 原本

Wikiページの原本は `docs/wiki/**/*.md` である。

GitHub上のWiki画面から直接編集すると、次回同期時に上書きされる可能性がある。通常はリポジトリ側のMarkdownを編集する。

## JSON索引

- `data/wiki-index.json` は `docs/wiki/**/*.md` から自動生成される派生索引である
- JSON索引は正本ではなく、直接編集してはならない
- 索引にはページタイトル、ファイルパス、分類、全文、コンテンツハッシュ、見出し、Wikiリンク、未作成リンク、バックリンクを格納する
- 索引は `scripts/build_wiki_index.py` によって決定的に生成する
- ページはタイトル順に整列し、生成時刻など不要な可変値を含めない
- `docs/wiki` または生成スクリプトを変更すると、GitHub Actionsが索引を再生成してコミットする
- ChatGPTは更新対象を探す際、まずJSON索引でページ名、リンク関係、候補ページを確認し、実際の更新前には必ず対象Markdownの最新版を読む
- JSON索引とMarkdownの内容が食い違う場合は、Markdownを正しいものとして扱い、索引を再生成する
- JSON索引だけを根拠にページを更新してはならない
- JSON索引に含まれる未作成リンクは、意図的な将来ページ候補を含むため、単純なリンク切れとして削除しない

## ページ名と配置

- 1ページにつき1つの主題を扱う
- 同じ意味のページを重複して作らない
- ページ名は短く、検索しやすい表現にする
- 人名ページは本人の同意と管理方針を確認してから作る
- ページはオントロジーを意識し、適切なサブフォルダへ配置する
- 1フォルダへ過度に集中させず、概念、ソフトウェア、ゲーム、人物、歴史、科学、資料などの分類を使い分ける
- フォルダは分類を表し、Wikiリンクは分類を越えた意味関係を表す

## 変更

- ChatGPTは既存ページから特徴的な単語を拾い、リンク先の雑学ページを追加してよい
- ランダムリンク先の雑学も、調査と出典確認を行ってから作る
- 大幅な削除や統合は、変更理由をコミットメッセージに残す
- ページ名変更時はリンク元も更新する
- ページ移動時はWikiリンク、JSON索引、生成スクリプト、GitHub Actions、Wiki同期を確認する
- 誤りを見つけた場合は、黙って残さず修正し、必要なら修正理由を記録する
- 更新後は文体、リンク切れ、重複、出典、プライバシーを横断確認する

## 後続ページ生成

- 後続処理は、JSON索引に記録された未作成リンクから優先的に新規ページを作る
- 現在は未作成リンクの総数を減らし、収束させることを優先する
- 深さ優先ではなく広さ優先で処理し、同じ枝から連続して下位概念を作らない
- 1回の処理では、参照元ページまたは概念群が異なる候補を原則3〜5件選んでページ化する
- `[[ボードゲームのメカニクス]]` から直接または間接に派生した候補を優先するが、ドラフト、セットコレクション、協力、競り、資源管理、勝利条件、情報構造など複数の枝へ分散させる
- 選定順位は、複数ページから参照される候補、既存ページとの接続性が高い候補、意味が明確な複合語、古くから残る候補の順とする
- 作成するページには、その概念を説明できるだけの内容を持たせ、単なる辞書的一文だけで終わらせない
- 収束期間中、新規ページから追加する未作成リンクは原則0〜1件、必要性が明確な場合でも最大2件とする
- 1回の処理で、作成したページ数より新たに追加した未作成リンク数を少なくする
- 新しい未作成リンクを増やすより、既存ページへのリンクを優先する
- 同義語、表記揺れ、過細分化、既存ページ内で説明可能な下位概念は、新規ページを作らず既存ページへの統合またはリンク修正で解消する
- 各処理の前後で未作成リンク総数を比較し、原則として純減していることを確認する

## 複合語のリンク化

- 本文中の特徴的な複合語、造語、専門用語、概念名は積極的に `[[Wikiリンク]]` 化する
- リンク先ページが存在しなくてもリンク化してよいが、収束期間中は必要性の低い新規未作成リンクを増やさない
- 「〜方式」「〜問題」「〜現象」「〜モデル」「〜設計」「〜仮説」「〜系」「〜論」「〜原則」「〜パターン」など、概念として独立し得る語句を優先する
- カタカナと漢字、英字と日本語などを組み合わせた特徴的な語句も優先する
- 一般的すぎる単語を無差別にリンクせず、ページとして説明を展開できるまとまりを選ぶ
- 同じページ内で同一語が繰り返される場合は、原則として最初の出現を優先してリンクする
- 見出し、コード、URL、既存リンクの内部は重ねてリンク化しない
- 長い複合語と短い語が重なる場合は、原則として長い複合語を優先する
- 未作成リンクは将来ページの候補として扱うが、候補数を無制限に増やすこと自体を目的としない

## プライバシー

- 趣味、制作テーマ、公開してよいと本人が指定した嗜好は記載してよい
- 家族、住所、勤務先、収入、健康状態など、私生活に踏み込む情報は本人の明示的な許可なく書かない

## リンク

GitHub Wiki形式のリンクを使用する。

    [[ページ名]]

表示名を変える場合は次の形式とする。

    [[表示名|ページ名]]

リンク先の `ページ名` には、見出しの表示名ではなく、対象Markdownのファイル名から拡張子 `.md` を除いた名前を指定する。サブフォルダは分類用であり、Wikiリンクでは原則としてファイル名だけを使う。

たとえば `docs/wiki/Home.md` の見出しが「SampleWiki」であっても、リンク先は `Home` である。

    [[SampleWiki|Home]]

`_Sidebar.md` ではリンク切れを許容しない。追加・変更するすべてのリンクについて、対応するMarkdownファイルが `docs/wiki` 配下に存在することを確認する。ページ名変更、移動、統合、削除の際は、本文だけでなく `_Sidebar.md` のリンクも同時に更新する。
